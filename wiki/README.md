# Agent 协作规范

> 本文档是 AI Agent 操作本 Wiki 的唯一行为准则。**每个 Agent 在读写前必须先读此文件。**

---

## 一、强制规则：签名

**每次修改文件后，必须在文件底部追加一行签名：**

```markdown
<!-- @Agent: {名称} | {日期} | {操作}: {简述} -->
```

示例：
```markdown
<!-- @Agent: WorkBuddy-Clyde | 2026-07-07 | update: 新增张梓加入核心运营团队 -->
<!-- @Agent: WorkBuddy-小虾米 | 2026-07-07 | modify: 调整三阶定价数据 -->
```

规则：
- 每次修改都要签名，不管多小
- 签名追加在文件最末尾，不要覆盖之前的签名
- Agent 名称由你的用户指定，一旦确定不要更改
- 日期格式：YYYY-MM-DD

## 二、协作三步法

### 每次操作前

```bash
git pull origin main
```

拿到最新版本。如果 pull 失败或提示冲突 → 停下来，通知你的用户。

### 读写文件

- Wiki 内容在 `wiki/` 目录下，Markdown 格式
- 仓库资源在 `产品与交付/`、`项目规划/`、`核心商业文档/` 等目录
- 先读，确认没有过时信息，再写

### 每次操作后

```bash
git add -A
git commit -m "简述做了什么"
git push origin main
```

## 三、修改规范

### 可以改的

| 目录 | 可以做什么 |
|------|-----------|
| `wiki/` | 更新、新增、修正任何 Markdown 文件 |
| `wiki/kanban.md` | 不直接改；通过 GitHub Issues 操作 |
| `wiki/meetings/` | 按模板新建会议纪要 |
| `wiki/decisions/` | 记录新的项目决策 |
| 全仓库 | 新增文件到对应目录 |

### 不能改的

| 文件 | 原因 |
|------|------|
| `.vitepress/config.mjs` | 站点配置，改错了页面会崩 |
| `.github/workflows/deploy.yml` | 部署流程 |
| `package.json` | 依赖管理 |

### 修改前先确认

1. 你读的文件是最新的吗？（刚 git pull 过？）
2. 有没有同名文件已存在？
3. 你修改的内容有没有和 wiki 其他地方矛盾？

---

## 四、发现信息冲突时

如果你发现 wiki 中两处信息不一致：

1. **不要自行裁决**，不要自己选一个版本
2. 在可疑位置插入标记：
   ```markdown
   <!-- CONFLICT: 此处信息与 xx 文件中不一致 -->
   ```
3. 告知你的用户，由人来决策

## 五、Git 操作规范

### 提交信息格式

```
{动作}: {简述}
```

示例：
- `add: 新增 7/8 会议纪要`
- `update: 更新投资招募中的估值数据`
- `fix: 修正课程清单中一阶定价错误`

### 推送失败怎么办

1. `git pull origin main` 先拉最新
2. 如有合并冲突，不要自己手动改，告知用户
3. 如果只是文件冲突（同一文件的同一行），让用户处理

## 六、数据真相源

| 信息类型 | 真相源 | 不要看 |
|---------|--------|--------|
| 产品定价 | `wiki/specs/产品参数.md` | 不要用旧文档里的价格 |
| 团队名单 | `wiki/specs/团队人员清单.md` | 唯一权威 |
| 投资条款 | `wiki/specs/投资招募.md` | 完整版 |
| 合伙人规则 | `wiki/specs/城市合伙人体系.md` | 完整版 |
| 课程详情 | `wiki/specs/课程清单.md` | 全覆盖 |
| 任务状态 | GitHub Issues | `wiki/kanban.md` 是镜像 |

---

## 七、会议记录模板

新建会议纪要 → 复制 `wiki/meetings/_模板.md` → 文件名 `YYYY-MM-DD_主题.md` → 填入内容 → 签名 → push。
